HBase测试

发表于 2018-11-30 | 分类于 hbase

hbase表有1028个region，5个regionserver，每个regionserver有200个region

每台regionserver分配8g内存。

写入数据：

​ 表中已有30g数据，一次性写入30g数据，12288条数据，每条数据2M大小。

​ 写入结束后，检查hdfs上该表大小 hadoop fs -du -s -h /hbase/data/default/baimu。

​ 可以发现，只刷写了15g的数据。因为还有15g数据在memstore中没有触发刷写。

检查hbase web ui 可以发现该表的每个region中的memstore只有5-10m。但此时总的memstore已经达到全局触发flush的阈值。flush的很频繁，并且flush的数据很小、

观察监控界面

分析：

​ 当前regionserver的分配可用内存为8G

​ hbase.regionserver.global.memstore.upperLimit/lowerLimit 默认值：0.4/0.35

​ hbase.hregion.memstore.flush.size 设置256Mb

​ region 1024个

​ 可以推算得出当该regionserver中memstore占用的总内存达到3g。hbase就会阻塞写操作，然后hbase会找一个最大的region进行刷写，释放memstore内存。如果当达到最高水位则会阻塞所有写操作。释放所有的region中memstore的内存。

​ 从图中发现，在10:2启动写入程序后到10:07之间的5分钟，该节点的每个region都没有刷写，由于rowkey的设计，每一个写请求均匀的落在每一个region上。即时当总memstore内存达到3g，然而每个region中的memstore可能只有15Mb。几乎不可能达到我们配置的 hbase.hregion.memstore.flush.size，就因为总的全局memstore限制而进行flush。

​ 10：10分时，memstore总内存3.1G,前1分钟写入请求210次。flush了18次，总共刷写到磁盘300Mb。估算一下得出每3s会flush一次，每次刷写16Mb。可见flush太过频繁，并且每次flush的数据量很小从而产生了大量小文件。后续hbase还需要minor compact来处理这些小文件增加压力。region的过多对集群性能和稳定性都有很大的影响。

​ 随后我们测试了一张只在一个regionserver上并且只有16个region的table。

先手动flush,将regionserver的memstore内存中上次实验的数据刷写到hdfs，释放memstore。 命令：flush ‘baimu’

下图是命令之行后，可以在HBase的web界面看到该表的每个region中memstore在释放。

分析：

​ 当前regionserver的分配可用内存为8G

​ hbase.regionserver.global.memstore.upperLimit/lowerLimit 默认值：0.4/0.35

​ hbase.hregion.memstore.flush.size 设置256Mb

​ region 8个

​ 3Gb/8=384Mb,也就是说总的memstore大小几乎不可能达到阈值，因为每个region中的memstore会应为先达到hbase.hregion.memstore.flush.size就被刷写。

​ 在11:22时memstore总内存大概是2Gb，前一分钟总共有900多次写请求（因为写压力都在这一台regionserver上，所以请求是上次实验的5倍），flush了13次，总共flush了3Gb多的数据。每次刷写236Mb左右的数据。符合我们的预期推算，也就是达到了我们设置的hbase.hregion.memstore.flush.size大小就开始刷写。这样每次刷写的文件就是256Mb。后期HBase会minor compact这些文件，生成多个稍大的文件，最后major compact成一个文件。

两次实验对比：

​

1Tb/5个节点=204.8Gb (每台regionserver的存储量)

​ 磁盘：

设置每个region file大小10Gb

​ 每个节点需要 204.8Gb/10Gb=20个region

​ 如果是5个regionserver，总共分100个region

​ 内存：

​ hbase heap0.4=8Gb0.4=3.2Gb

3.2Gb/memstoresize=(memstoresize我按设置的256Mb算的)合理的region个数。

大概如此估算:

比如10TB数据。内存8Gb。Memstore256Mb。

3.2Gb/256Mb=13个region

10Tb/5个节点=2Tb

2Tb/13=160Gb

也就是说当前集群情况如果需要写入10Tb的数据最好是每个region大小设置为160Gb。每个regionserver的region个数是13个

如果memstoresize设置128Mb

3.2Gb/128Mb=26个region

10Tb/5个节点=2Tb

2Tb/26=80Gb

此情况下要写入10Tb的数据最好是每个region大小设置为80Gb。每个regionserver的region个数是26个。

可以调节的参数 有hbase可使用内存、memstoresize、regionsize、region个数。
